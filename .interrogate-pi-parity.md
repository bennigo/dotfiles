# Destination: Pi ↔ Claude Code Parity Setup

## Problem

Pi is installed and configured but lacks the capability stack that makes Claude Code productive:
web search, postgres database access, sub-agents, memory persistence, and vault integration.
Without these, ~10 of 40 shared skills fail at runtime and daily coding workflows (database
queries, web research, multi-agent delegation) require switching back to Claude Code.
The goal is **not** to replicate Claude Code's MCP architecture but to build a Pi-native
equivalent using Pi's primitives (extensions, skills, sub-agents) that shares the same
skill/memory/context ecosystem and can realistically handle a full day of work.

## In scope

1. **Web search + fetch extension** — TypeScript extension wrapping Brave Search API with
   structured results and parallel page fetching. Unblocks 10 MCP-dependent skills.
2. **Postgres extension** — Hybrid tool: auto-schema introspection (`\dt` + `\d+`) before
   every query, `EXPLAIN` validation, read-write gate for `postgres-local`. Accesses all
   8 databases currently in `~/.mcp.json`.
3. **Sub-agent extension** — Built-in `subagent/` example, installed and customized with
   agent definitions for scout (cheap model), planner (premium), reviewer, worker.
4. **Agent definitions** — `~/.pi/agent/agents/*.md` mapping your Claude Code agent roles
   to Pi sub-agents, with per-agent model selection (Copilot for flat-rate, DeepSeek/Ollama
   for cheap, Sonnet for quality).
5. **Pi-native web/scholar search skill** — Lightweight bridge skill so the 10 broken
   shared skills can resolve through Pi's web extension.
6. **Memory bridge** — Pi writes to `~/.claude/projects/<encoded-cwd>/memory/` (same tree
   Claude Code uses). Append-only, shared read pool.
7. **Vault integration skill** — Wraps `vault-lookup.py` and `learning-query.py` for Pi.
8. **Context-lean strategy** — Session branching + compaction + per-agent tool sets to keep
   context under 60% during long sessions.
9. **Capture pipeline** — `/capture` extension command → drops ideas into `0.Inbox/` for
   later sorting. Prevents lost mid-session ideas.

## Out of scope

- **Google Workspace integration** — Backburner; revisit after permission boundary testing.
- **Replicating MCP layer** — Wrong architecture. Pi extensions talk to services directly.
- **Rewriting 30 working shared skills** — They work and stay shared. Only the 10 MCP-dependent
  skills get Pi-native alternatives (via web search extension).
- **Auto-memory black-box** — Memory bridge is preferred; Pi sessions are transparent JSONL.
- **Changing vault/CLAUDE.md hierarchy** — Shared infrastructure is working well.
- **Unix-philosophy skill refactor** — Flagged as future initiative, not this round.

## Architectural decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Extension organization | Single Pi package (`bgo-toolkit`) under dotfiles | Stow-deployable, git-backed, follows existing pattern |
| Postgres access | Hybrid: auto-introspect + SQL passthrough + EXPLAIN gate | Never caches schema; always queries live `\dt`/`\d+` |
| Web search | TypeScript extension (not bash skill) | Structured JSON parsing, parallel fetch, cleaner LLM synthesis |
| Sub-agents | Built-in `subagent/` example (process isolation) | Battle-tested, per-agent model selection, Copilot flat-rate |
| Encryption | None needed | Secrets via `!pass show` references, not literal values |
| Memory bridge | Write to `~/.claude/projects/` tree (append-only) | Shared read pool, no merge conflicts |

## Architecture principle (future)

Context budget per project should scale sub-linearly with project count. Not in litmus test
for this round, but design choices (session branching, per-agent tools, no schema caching)
support this direction.

## Downstream

**Recommended: Install + configure first, then build.** Directory scaffolding, stow symlinks,
package installs (~10 min) → then build extensions one at a time: postgres first (heaviest
dependency), then web search, then subagent, then bridge skills.
